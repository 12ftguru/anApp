What is anApp?

anApp is an Application Framework for PHP and Javascript that focuses on providing utility while trying to stay out of the programmer's way as much as possible.

anApp is comprised of four parts:

  • aLib – the core low-level php libraries, which may be used separately from the other parts of anApp
  • anApp PHP Backend – the PHP backend framework, depends on aLib, but may be used independently from anApp JS
  • anApp JS – the Javascript ui framework, may be used independently from anApp PHP but requires the Ext JS javascript library.
  • anApp-based functionality modules – Reusable bits of code that actually form the functionality of the application. These are written by you, by us and by the community but are not part of the actual anApp distribution.

What makes anApp different?

anApp was created with several underlying philosophies:

Get out of the developer's way

In these docs, and in the comments in the code, you'll find phrases like "if you want to use this" or "you don't have to use this, but…" I know when I'm coding, I'm fairly particular about how I like to do things. I want the required parts of anApp to be as few in number as possible. No one should have to use any component they don't want to. Don't want to use the iform object? Don't! I could care less.

Thus every bit of anApp, with a very few exceptions, should be discrete and optional. The various modules you build your application out of may have requirements and dependencies, but that's a different story. And they should be fairly pluggable.

Allow the developer to do as little work as possible

Ideally, creating a new anApp-based application will involve copying the basic framework files to a new directory, choosing modules from the library, copying those files into your new application directory and tweaking them. I know that it will rarely work out that way, but writing one or two new modules per application is better than starting from scratch.

One thing does one thing only

If a file is called "saveNote.inc" that file really should just save notes (and reply with the success/failure of said save). It shouldn't be used to save or display anything else. Sure, this makes for lots of little files lying around, but if you want to know what http://myApp/saveNote does, you've got a pretty good idea of where to find out.

Why 3.0? What happened to versions 1 and 2?

Version 1.0 of anApp was our original, in-house version. It is running on sites like http://www.12ftguru.com/ and http://www.thoughtholder.com/ and http://www.bandmetrics.com/ That version was using version 2.1 of aLib. Since aLib is going to version 3 in this release, I thought it would be confusing to put anApp at version 2. So everything is version 3, which means we get to bypass the terrible twos.

The initial commit to Google Code is our 3.0 pre-release. The following are left to be completed to meet our 3.0 milestone:

Rules of the Road, an anApp Style Guide

Naming Conventions

I prefer camelCase over under_scores, although there's little rationale behind it. I just like camelCase better. Any object that extends anObject (including all of the aLib objects) will automatically convert between the two for methods: 

 1:  
 2:  class swissArmyKnife extends anObject {
 3:  
 4:     public function bottleOpener() {
 5:        print "Works";
 6:     }
 7:  
 8:     public function tooth_pick() {
 9:        print "I lost it again.";
10:     }
11:  
12:  }
13:  
14:  $myObj = new swissArmyKnife();
15:  
16:  $myObj->bottleOpener(); //Works
17:  $myObj->bottle_opener(); //Works
18:

I should probably do the same for class attributes.

So the point is kind of moot, but I find that reading code is easier if everything looks the same. So… naming conventions I follow:

Database

  • No underscores
  • unique autoincrement ids are named <singular tablename>ID such that a table called "things" would have a unique ID thingID
    The thinking is that a row of a table is a singluar instance, so a table containing things would have rows that each represent one thing.
  • table names are plural, lower case, alphanumeric characters only
    Good: mythings Bad: myThings Bad: mythings Bad: mything Bad: somethinghavingnothingtodowithmything

    Note, the corresponding ID would be myThingID

    • Raw data tables should have the word "raw" as the first part of the table name.
  • Column names are alphanumeric mixed case, with words camelcased

PHP

  • Classes
    • One class per file
      • Filename is classname.inc
      • If the file has an accompanying html template:
        • Single templates should be called classname.ihtml
        • Multiple template files should be named classname-<some descriptor>.ihtml
  • Class, function and variable names are camelcased, alphanumeric characters only
